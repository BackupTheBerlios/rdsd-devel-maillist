schrieb "Daniel Lindenaar (rdsd)" <daniel-rdsd at lindenaar.eu>:

> Hi guys,
> Another thing I came across: rdsd often quits unexpectedly,
> especially when stopping and starting rdsquery a lot. It seems to be
> caused by a SIGPIPE signal that's thrown when rdsd tries to write to
> a disconnected client. I've added SIGPIPE to the list of signals that
> get caught and now rdsd doesn't quit anymore. Any ideas on this?
> maybe this is the way to go or maybe there's something that needs to
> be done in the client handling code.

Hmm, I never saw that issue. rdsd does all communication through
sockets, and you can usually detect a disconnected client by checking
the return value of your read/write functions. That's how it's done in
rdsd. 
Did you test with TCP/IP or unix domain sockets? I wasn't aware that
you can receive a SIGPIPE when a socket connection breaks. Seems I
have to read some basic docs again :-)
Anyway, if rdsd receives SIGPIPE signals, it is of course the right
thing to catch them. It shouldn't need special handling, though. I
tested that quite a lot and think that disconnecting clients are
detected reliably. OTOH, I only tested with clients that disconnect
gracefully. Did you just pull the cable?
